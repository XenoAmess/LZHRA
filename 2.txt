<p>The <b>Lempel–Ziv–Markov chain algorithm</b> (<b>LZMA</b>) is an <a href="/wiki/Algorithm" title="Algorithm">algorithm</a> used to perform <a href="/wiki/Lossless_data_compression" class="mw-redirect" title="Lossless data compression">lossless data compression</a>. It has been under development either since 1996 or 1998<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup> and was first used in the <a href="/wiki/7z" title="7z">7z</a> format of the 7-Zip archiver. This algorithm uses a <a href="/wiki/Dictionary_coder" title="Dictionary coder">dictionary compression</a> scheme somewhat similar to the <a href="/wiki/LZ77" class="mw-redirect" title="LZ77">LZ77</a> algorithm published by <a href="/wiki/Abraham_Lempel" title="Abraham Lempel">Abraham Lempel</a> and <a href="/wiki/Jacob_Ziv" title="Jacob Ziv">Jacob Ziv</a> in 1977 and features a high compression ratio (generally higher than <a href="/wiki/Bzip2" title="Bzip2">bzip2</a>)<sup id="cite_ref-gzip_bzip_lzma_bench_2-0" class="reference"><a href="#cite_note-gzip_bzip_lzma_bench-2">[2]</a></sup><sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup> and a variable compression-dictionary size (up to 4&#160;<a href="/wiki/Gigabyte" title="Gigabyte">GB</a>),<sup id="cite_ref-LZMA_Format_4-0" class="reference"><a href="#cite_note-LZMA_Format-4">[4]</a></sup> while still maintaining decompression speed similar to other commonly used compression algorithms.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup></p>
<p><b>LZMA2</b> is a simple container format that can include both uncompressed data and LZMA data, possibly with multiple different LZMA encoding parameters. LZMA2 supports arbitrarily scalable multithreaded compression and decompression and efficient compression of data which is partially incompressible.</p>

<p>LZMA uses a dictionary compression algorithm (a variant of <a href="/wiki/LZ77" class="mw-redirect" title="LZ77">LZ77</a> with huge dictionary sizes and special support for repeatedly used match distances), whose output is then encoded with a <a href="/wiki/Range_encoding" title="Range encoding">range encoder</a>, using a complex model to make a probability prediction of each bit. The dictionary compressor finds matches using sophisticated dictionary data structures, and produces a stream of literal symbols and phrase references, which is encoded one bit at a time by the range encoder: many encodings are possible, and a <a href="/wiki/Dynamic_programming" title="Dynamic programming">dynamic programming</a> algorithm is used to select an optimal one under certain approximations.</p>
<p>Prior to LZMA, most encoder models were purely byte-based (i.e. they coded each bit using only a cascade of contexts to represent the dependencies on previous bits from the same byte). The main innovation of LZMA is that instead of a generic byte-based model, LZMA's model uses contexts specific to the bitfields in each representation of a literal or phrase: this is nearly as simple as a generic byte-based model, but gives much better compression because it avoids mixing unrelated bits together in the same context. Furthermore, compared to classic dictionary compression (such as the one used in <a href="/wiki/ZIP_(file_format)" class="mw-redirect" title="ZIP (file format)">zip</a> and <a href="/wiki/Gzip" title="Gzip">gzip</a> formats), the dictionary sizes can be and usually are much larger, taking advantage of the large amount of memory available on modern systems.</p>

<p>In LZMA compression, the compressed stream is a stream of bits, encoded using an adaptive binary range coder. The stream is divided into packets, each packet describing either a single byte, or an LZ77 sequence with its length and distance implicitly or explicitly encoded. Each part of each packet is modeled with independent contexts, so the probability predictions for each bit are correlated with the values of that bit (and related bits from the same field) in previous packets of the same type.</p>

<p>No complete natural language specification of the compressed format seems to exist, other than the one attempted in the following text.</p>
<p>The description below is based on the compact <a href="/wiki/Xz" title="Xz">XZ</a> Embedded decoder by Lasse Collin included in the Linux kernel source<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup> from which the LZMA and LZMA2 algorithm details can be relatively easily deduced: thus, while citing source code as reference isn't ideal, any programmer should be able to check the claims below with a few hours of work.</p>
<h3><span class="mw-headline" id="Range_coding_of_bits">Range coding of bits</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=4" title="Edit section: Range coding of bits">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>LZMA data is at the lowest level decoded one bit at a time by the range decoder, at the direction of the LZMA decoder.</p>
<p>Context-based range decoding is invoked by the LZMA algorithm passing it a reference to the "context", which consists of the unsigned 11-bit variable <i>prob</i> (typically implemented using a 16-bit data type) representing the predicted probability of the bit being 1, which is read and updated by the range decoder (and should be initialized to 2^10, representing 0.5 probability).</p>
<p>Fixed probability range decoding instead assumes a 0.5 probability, but operates slightly differently from context-based range decoding.</p>
<p>The range decoder state consists of two unsigned 32-bit variables, <i>range</i> (representing the range size), and <i>code</i> (representing the encoded point within the range).</p>
<p>Initialization of the range decoder consists of setting <i>range</i> to 2<sup>32</sup>&#160;−&#160;1, and <i>code</i> to the 32-bit value starting at the second byte in the stream interpreted as big-endian; the first byte in the stream is completely ignored.</p>
<p>Normalization proceeds in this way:</p>
<ol>
<li>Shift both <i>range</i> and <i>code</i> left by 8 bits</li>
<li>Read a byte from the compressed stream</li>
<li>Set the least significant 8 bits of <i>code</i> to the byte value read</li>
</ol>
<p>Context-based range decoding of a bit using the <i>prob</i> probability variable proceeds in this way:</p>
<ol>
<li>If <i>range</i> is less than 2^24, perform normalization</li>
<li>Set <i>bound</i> to floor(<i>range</i> / 2^11) * <i>prob</i></li>
<li>If <i>code</i> is less than <i>bound</i>:
<ol>
<li>Set <i>range</i> to <i>bound</i></li>
<li>Set <i>prob</i> to <i>prob</i> + floor((2^11 - <i>prob</i>) / 2^5)</li>
<li>Return bit 0</li>
</ol>
</li>
<li>Otherwise (if <i>code</i> is greater than or equal to the <i>bound</i>):
<ol>
<li>Set <i>range</i> to <i>range</i> - <i>bound</i></li>
<li>Set <i>code</i> to <i>code</i> - <i>bound</i></li>
<li>Set <i>prob</i> to <i>prob</i> - floor(<i>prob</i> / 2^5)</li>
<li>Return bit 1</li>
</ol>
</li>
</ol>
<p>Fixed-probability range decoding of a bit proceeds in this way:</p>
<ol>
<li>If <i>range</i> is less than 2^24, perform normalization</li>
<li>Set <i>range</i> to floor(<i>range</i> / 2)</li>
<li>If <i>code</i> is less than <i>range</i>:
<ol>
<li>Return bit 0</li>
</ol>
</li>
<li>Otherwise (if <i>code</i> is greater or equal than <i>range</i>):
<ol>
<li>Set <i>code</i> to <i>code</i> - <i>range</i></li>
<li>Return bit 1</li>
</ol>
</li>
</ol>
<p>The Linux kernel implementation of fixed-probability decoding in rc_direct, for performance reasons, doesn't include a conditional branch, but instead subtracts <i>range</i> from <i>code</i> unconditionally, and uses the resulting sign bit to both decide the bit to return, and to generate a mask that is combined with <i>code</i> and added to <i>range</i>.</p>
<p>Note that:</p>
<ol>
<li>The division by 2^11 when computing <i>bound</i> and floor operation is done before the multiplication, not after (apparently to avoid requiring fast hardware support for 32-bit multiplication with a 64-bit result)</li>
<li>Fixed probability decoding is not strictly equivalent to context-based range decoding with any <i>prob</i> value, due to the fact that context-based range decoding discards the lower 11 bits of <i>range</i> before multiplying by <i>prob</i> as just described, while fixed probability decoding only discards the last bit</li>
</ol>
<h3><span class="mw-headline" id="Range_coding_of_integers">Range coding of integers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=5" title="Edit section: Range coding of integers">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The range decoder also provides the bit-tree, reverse bit-tree and fixed probability integer decoding facilities, which are used to decode integers, and generalize the single-bit decoding described above. To decode unsigned integers less than <i>limit</i>, an array of (<i>limit</i>&#160;−&#160;1) 11-bit probability variables is provided, which are conceptually arranged as the internal nodes of a complete binary tree with <i>limit</i> leaves.</p>
<p>Non-reverse bit-tree decoding works by keeping a pointer to the tree of variables, which starts at the root. As long as the pointer doesn't point to a leaf, a bit is decoded using the variable indicated by the pointer, and the pointer is moved to either the left or right children depending on whether the bit is 0 or 1; when the pointer points to a leaf, the number associated with the leaf is returned.</p>
<p>Non-reverse bit-tree decoding thus happens from most significant to least significant bit, stopping when only one value in the valid range is possible (this conceptually allows to have range sizes that are not powers of two, even though LZMA doesn't make use of this).</p>
<p>Reverse bit-tree decoding instead decodes from least significant bit to most significant bits, and thus only supports ranges that are powers of two, and always decodes the same number of bits. It is equivalent to performing non-reverse bittree decoding with a power of two <i>limit</i>, and reversing the last log2(<i>limit</i>) bits of the result.</p>
<p>Note that in the rc_bittree function in the Linux kernel, integers are actually returned in the [<i>limit</i>, 2 * <i>limit</i>) range (with <i>limit</i> added to the conceptual value), and the variable at index 0 in the array is unused, while the one at index 1 is the root, and the left and right children indices are computed as 2<i>i</i> and 2<i>i</i> + 1. The rc_bittree_reverse function instead adds integers in the [0, <i>limit</i>) range to a caller-provided variable, where <i>limit</i> is implicitly represented by its logarithm, and has its own independent implementation for efficiency reasons.</p>
<p>Fixed probability integer decoding simply performs fixed probability bit decoding repeatedly, reading bits from the most to the least significant.</p>
<h3><span class="mw-headline" id="LZMA_configuration">LZMA configuration</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=6" title="Edit section: LZMA configuration">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The LZMA decoder is configured by an <i>lclppb</i> "properties" byte and a dictionary size.</p>
<p>The value of the <i>lclppb</i> byte is <i>lc</i> + <i>lp</i> * 9 + <i>pb</i> * 9 * 5, where:</p>
<ul>
<li><i>lc</i> is the number of high bits of the previous byte to use as a context for literal encoding (the default value used by the LZMA SDK is 3)</li>
<li><i>lp</i> is the number of low bits of the dictionary position to include in <i>literal_pos_state</i> (the default value used by the LZMA SDK is 0)</li>
<li><i>pb</i> is the number of low bits of the dictionary position to include in <i>pos_state</i> (the default value used by the LZMA SDK is 2)</li>
</ul>
<p>In non-LZMA2 streams, <i>lc</i> must not be greater than 8, and <i>lp</i> and <i>pb</i> must not be greater than 4. In LZMA2 streams, (<i>lc</i> + <i>lp</i>) and <i>pb</i> must not be greater than 4.</p>
<p>In the 7-zip LZMA file format, configuration is performed by a header containing the "properties" byte followed by the 32-bit little-endian dictionary size in bytes. In LZMA2, the properties byte can optionally be changed at the start of LZMA2 LZMA packets, while the dictionary size is specified in the LZMA2 header as later described.</p>
<h3><span class="mw-headline" id="LZMA_coding_contexts">LZMA coding contexts</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=7" title="Edit section: LZMA coding contexts">edit</a><span class="mw-editsection-bracket">]</span></span></h3>

<p>The <i>pos_state</i> and <i>literal_pos_state</i> values consist of respectively the <i>pb</i> and <i>lp</i> (up to 4, from the LZMA header or LZMA2 properties packet) least significant bits of the dictionary position (the number of bytes coded since the last dictionary reset modulo the dictionary size). <p>The <i>prev_byte_lc_msbs</i> value is set to the <i>lc</i> (up to 4, from the LZMA header or LZMA2 properties packet) most significant bits of the previous uncompressed byte.</p>
<p>The <i>is_REP</i> value denotes whether a packet that includes a length is a LONGREP rather than a MATCH.</p>
<p>The <i>match_byte</i> value is the byte that would have been decoded if a SHORTREP packet had been used (in other words, the byte found at the dictionary at the last used distance); it is only used just after a *MATCH packet.</p>
<p><i>literal_bit_mode</i> is an array of 8 values in the 0-2 range, one for each bit position in a byte, which are 1 or 2 if the previous packet was a *MATCH and it is either the most significant bit position or all the more significant bits in the literal to encode/decode are equal to the bits in the corresponding positions in <i>match_byte</i>, while otherwise it is 0; the choice between the 1 or 2 values depends on the value of the bit at the same position in <i>match_byte</i>.</p>
<p>The literal/Literal set of variables can be seen as a "pseudo-bit-tree" similar to a bit-tree but with 3 variables instead of 1 in every node, chosen depending on the <i>literal_bit_mode</i> value at the bit position of the next bit to decode after the bit-tree context denoted by the node.</p>

<h3><span class="mw-headline" id="LZMA2_format">LZMA2 format</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=8" title="Edit section: LZMA2 format">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The LZMA2 container supports multiple runs of compressed LZMA data and uncompressed data. Each LZMA compressed run can have a different LZMA configuration and dictionary. This improves the compression of partially or completely incompressible files and allows multithreaded compression and multithreaded decompression by breaking the file into runs that can be compressed or decompressed independently in parallel.</p>
<p>The LZMA2 header consists of a byte indicating the dictionary size:</p>
<ul>
<li>40 indicates a 4 GB − 1 dictionary size</li>
<li>Even values less than 40 indicate a 2<sup><i>v</i>/2 + 12</sup> bytes dictionary size</li>
<li>Odd values less than 40 indicate a 3×2<sup>(<i>v</i> − 1)/2 + 11</sup> bytes dictionary size</li>
<li>Values higher than 40 are invalid</li>
</ul>
<p>LZMA2 data consists of packets starting with a control byte, with the following values:</p>
<ul>
<li>0 denotes the end of the file</li>
<li>1 denotes a dictionary reset followed by an uncompressed chunk</li>
<li>2 denotes an uncompressed chunk without a dictionary reset</li>
<li>3-0x7f are invalid values</li>
<li>0x80-0xff denotes an LZMA chunk, where the lowest 5 bits are used as bit 16-20 of the uncompressed size minus one, and bit 5-6 indicates what should be reset</li>
</ul>
<p>Bits 5-6 for LZMA chunks can be:</p>
<ul>
<li>0: nothing reset</li>
<li>1: state reset</li>
<li>2: state reset, properties reset using properties byte</li>
<li>3: state reset, properties reset using properties byte, dictionary reset</li>
</ul>
<p>LZMA state resets cause a reset of all LZMA state except the dictionary, and specifically:</p>
<ul>
<li>The range coder</li>
<li>The <i>state</i> value</li>
<li>The last distances for repeated matches</li>
<li>All LZMA probabilities</li>
</ul>
<p>Uncompressed chunks consist of:</p>
<ul>
<li>A 16-bit big-endian value encoding the data size minus one</li>
<li>The data to be copied verbatim into the dictionary and the output</li>
</ul>
<p>LZMA chunks consist of:</p>
<ul>
<li>A 16-bit big-endian value encoding the low 16-bits of the uncompressed size minus one</li>
<li>A 16-bit big-endian value encoding the compressed size minus one</li>
<li>A properties/lclppb byte if bit 6 in the control byte is set</li>
<li>The LZMA compressed data, starting with the 5 bytes (of which the first is ignored) used to initialize the range coder (which are included in the compressed size)</li>
</ul>
<h3><span class="mw-headline" id="xz_and_7z_formats">xz and 7z formats</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=9" title="Edit section: xz and 7z formats">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The .[xz] format, which can contain LZMA2 data, is documented at <i>tukaani.org</i>,<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup> while the .7z file format, which can contain either LZMA or LZMA2 data, is documented in the 7zformat.txt file contained in the LZMA SDK.<sup id="cite_ref-LZMA_SDK_8-0" class="reference"><a href="#cite_note-LZMA_SDK-8">[8]</a></sup></p>
<h2><span class="mw-headline" id="Compression_algorithm_details">Compression algorithm details</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=10" title="Edit section: Compression algorithm details">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Similar to the decompression format situation, no complete natural language specification of the encoding techniques in <a href="/wiki/7-zip" class="mw-redirect" title="7-zip">7-zip</a> or <a href="/wiki/Xz" title="Xz">xz</a> seems to exist, other than the one attempted in the following text.</p>
<p>The description below is based on the <a href="/wiki/XZ_Utils" title="XZ Utils">XZ</a> for Java encoder by Lasse Collin,<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup> which appears to be the most readable among several rewrites of the original 7-zip using the same algorithms: again, while citing source code as reference isn't ideal, any programmer should be able to check the claims below with a few hours of work.</p>
<h3><span class="mw-headline" id="Range_encoder">Range encoder</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=11" title="Edit section: Range encoder">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The range encoder cannot make any interesting choices, and can be readily constructed based on the decoder description.</p>
<p>Initialization and termination are not fully determined; the <a href="/wiki/Xz" title="Xz">xz</a> encoder outputs 0 as the first byte which is ignored by the decompressor, and encodes the lower bound of the range (which matters for the final bytes).</p>
<p>The <a href="/wiki/Xz" title="Xz">xz</a> encoder uses an unsigned 33-bit variable called <i>low</i> (typically implemented as a 64-bit integer, initialized to 0), an unsigned 32-bit variable called <i>range</i> (initialized to 2<sup>32</sup>&#160;−&#160;1), an unsigned 8-bit variable called <i>cache</i> (initialized to 0), and an unsigned variable called <i>cache_size</i> which needs to be large enough to store the uncompressed size (initialized to 1, typically implemented as a 64-bit integer).</p>
<p>The <i>cache</i>/<i>cache_size</i> variables are used to properly handle carries, and represent a number defined by a big-endian sequence starting with the <i>cache</i> value, and followed by <i>cache_size</i> 0xff bytes, which has been shifted out of the <i>low</i> register, but hasn't been written yet, because it could be incremented by one due to a carry.</p>
<p>Note that the first byte output will always be 0 due to the fact that <i>cache</i> and <i>low</i> are initialized to 0, and the encoder implementation; the <a href="/wiki/Xz" title="Xz">xz</a> decoder ignores this byte.</p>
<p>Normalization proceeds in this way:</p>
<ol>
<li>If <i>low</i> is less than (2^32 - 2^24):
<ol>
<li>Output the byte stored in <i>cache</i> to the compressed stream</li>
<li>Output <i>cache_size</i> - 1 bytes with 0xff value</li>
<li>Set <i>cache</i> to bits 24-31 of <i>low</i></li>
<li>Set <i>cache_size</i> to 0</li>
</ol>
</li>
<li>If <i>low</i> is greater or equal than 2^32:
<ol>
<li>Output the byte stored in <i>cache</i> plus one to the compressed stream</li>
<li>Output <i>cache_size</i> - 1 bytes with 0 value</li>
<li>Set <i>cache</i> to bits 24-31 of <i>low</i></li>
<li>Set <i>cache_size</i> to 0</li>
</ol>
</li>
<li>Increment <i>cache_size</i></li>
<li>Set <i>low</i> to the lowest 24 bits of <i>low</i> shifted left by 8 bits</li>
<li>Set <i>range</i> to <i>range</i> shifted left by 8 bits</li>
</ol>
<p>Context-based range encoding of a bit using the <i>prob</i> probability variable proceeds in this way:</p>
<ol>
<li>If <i>range</i> is less than 2^24, perform normalization</li>
<li>Set <i>bound</i> to floor(<i>range</i> / 2^11) * <i>prob</i></li>
<li>If encoding a 0 bit:
<ol>
<li>Set <i>range</i> to <i>bound</i></li>
<li>Set <i>prob</i> to <i>prob</i> + floor((2^11 - <i>prob</i>) / 2^5)</li>
</ol>
</li>
<li>Otherwise (if encoding a 1 bit):
<ol>
<li>Set <i>range</i> to <i>range</i> - <i>bound</i></li>
<li>Set <i>low to</i> low + <i>bound</i></li>
<li>Set <i>prob</i> to <i>prob</i> - floor(<i>prob</i> / 2^5)</li>
</ol>
</li>
</ol>
<p>Fixed-probability range encoding of a bit proceeds in this way:</p>
<ol>
<li>If <i>range</i> is less than 2^24, perform normalization</li>
<li>Set <i>range</i> to floor(<i>range</i> / 2)</li>
<li>If encoding a 1 bit:
<ol>
<li>Set <i>low</i> to <i>low</i> + <i>range</i></li>
</ol>
</li>
</ol>
<p>Termination proceeds this way:</p>
<ol>
<li>Perform normalization 5 times</li>
</ol>
<p>Bit-tree encoding is performed like decoding, except that bit values are taken from the input integer to be encoded rather than from the result of the bit decoding functions.</p>
<p>For algorithms that try to compute the encoding with the shortest post-range-encoding size, the encoder also needs to provide an estimate of that.</p>
<h3><span class="mw-headline" id="Dictionary_search_data_structures">Dictionary search data structures</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=12" title="Edit section: Dictionary search data structures">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The encoder needs to be able to quickly locate matches in the dictionary. Since LZMA uses very large dictionaries (potentially on the order of gigabytes) to improve compression, simply scanning the whole dictionary would result in an encoder too slow to be practically usable, so sophisticated data structures are needed to support fast match searches.</p>
<h4><span class="mw-headline" id="Hash_chains">Hash chains</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=13" title="Edit section: Hash chains">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The simplest approach, called "hash chains", is parameterized by a constant N which can be either 2, 3 or 4, which is typically chosen so that 2^(8×<var style="padding-right: 1px;">N</var>) is greater than or equal to the dictionary size.</p>
<p>It consists of creating, for each <i>k</i> less than <i>N</i>, a hash table indexed by tuples of <i>k</i> bytes, where each of the buckets contains the last position where the first <i>k</i> bytes hashed to the hash value associated with that hash table bucket.</p>
<p>Chaining is achieved by an additional array which stores, for every dictionary position, the last seen previous position whose first <i>N</i> bytes hash to the same value of the first <i>N</i> bytes of the position in question.</p>
<p>To find matches of length <i>N</i> or higher, a search is started using the <i>N</i>-sized hash table, and continued using the hash chain array; the search stop after a pre-defined number of hash chain nodes has been traversed, or when the hash chains "wraps around", indicating that the portion of the input that has been overwritten in the dictionary has been reached.</p>
<p>Matches of size less than <i>N</i> are instead found by simply looking at the corresponding hash table, which either contains the latest such match, if any, or a string that hashes to the same value; in the latter case, the encoder won't be able to find the match. This issue is mitigated by the fact that for distant short matches using multiple literals might require less bits, and having hash conflicts in nearby strings is relatively unlikely; using larger hash tables or even direct lookup tables can reduce the problem at the cost of higher cache miss rate and thus lower performance.</p>
<p>Note that all matches need to be validated to check that the actual bytes match currently at that specific dictionary position match, since the hashing mechanism only guarantees that at some past time there were characters hashing to the hash table bucket index (some implementations may not even guarantee that, because they don't initialize the data structures).</p>
<p>LZMA uses <a href="/wiki/Markov_chains" class="mw-redirect" title="Markov chains">Markov chains</a>, as implied by "M" in its name.</p>
<h4><span class="mw-headline" id="Binary_trees">Binary trees</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=14" title="Edit section: Binary trees">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The <a href="/wiki/Binary_tree" title="Binary tree">binary tree</a> approach follows the hash chain approach, except that it logically uses a binary tree instead of a linked list for chaining.</p>
<p>The binary tree is maintained so that it is always both a search tree relative to the suffix lexicographic ordering, and a max-heap for the dictionary position<sup id="cite_ref-solomon_10-0" class="reference"><a href="#cite_note-solomon-10">[10]</a></sup> (in other words, the root is always the most recent string, and a child cannot have been added more recently than its parent): assuming all strings are lexicographically ordered, these conditions clearly uniquely determine the binary tree (this is trivially provable by induction on the size of the tree).</p>
<p>Since the string to search for and the string to insert are the same, it is possible to perform both dictionary search and insertion (which requires to rotate the tree) in a single tree traversal.</p>
<h4><span class="mw-headline" id="Patricia_tries">Patricia tries</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=15" title="Edit section: Patricia tries">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Some old LZMA encoders also supported a data structure based on <a href="/wiki/Patricia_trie" class="mw-redirect" title="Patricia trie">Patricia tries</a>, but such support has since been dropped since it was deemed inferior to the other options.<sup id="cite_ref-solomon_10-1" class="reference"><a href="#cite_note-solomon-10">[10]</a></sup></p>
<h3><span class="mw-headline" id="LZMA_encoder">LZMA encoder</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=16" title="Edit section: LZMA encoder">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>LZMA encoders can freely decide which match to output, or whether to ignore the presence of matches and output literals anyway.</p>
<p>The ability to recall the 4 most recently used distances means that, in principle, using a match with a distance that will be needed again later may be globally optimal even if it is not locally optimal, and as a result of this, optimal LZMA compression probably requires knowledge of the whole input and might require algorithms too slow to be usable in practice.</p>
<p>Due to this, practical implementations tend to employ non-global heuristics.</p>
<p>The <a href="/wiki/Xz" title="Xz">xz</a> encoders use a value called <i>nice_len</i> (the default is 64): when any match of length at least <i>nice_len</i> is found, the encoder stops the search and outputs it, with the maximum matching length.</p>
<h4><span class="mw-headline" id="Fast_encoder">Fast encoder</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=17" title="Edit section: Fast encoder">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The XZ fast encoder <sup id="cite_ref-11" class="reference"><a href="#cite_note-11">[11]</a></sup> (derived from the 7-zip fast encoder) is the shortest LZMA encoder in the <a href="/wiki/Xz" title="Xz">xz</a> source tree.</p>
<p>It works like this:</p>
<ol>
<li>Perform combined search and insertion in the dictionary data structure</li>
<li>If any repeated distance matches with length at least <i>nice_len</i>:
<ul>
<li>Output the most frequently used such distance with a REP packet</li>
</ul>
</li>
<li>If a match was found of length at least <i>nice_len</i>:
<ul>
<li>Output it with a MATCH packet</li>
</ul>
</li>
<li>Set the main match to the longest match</li>
<li>Look at the nearest match of every length in decreasing length order, and until no replacement can be made:
<ul>
<li>Replace the main match with a match which is one character shorter, but whose distance is less than 1/128 the current main match distance</li>
</ul>
</li>
<li>Set the main match length to 1 if the current main match is of length 2 and distance at least 128</li>
<li>If a repeated match was found, and its length is at least 1 character less than the main match:
<ul>
<li>Output the repeated match with a REP packet</li>
</ul>
</li>
<li>If a repeated match was found, and its length is at least 2 character less than the main match, and the main match distance is at least 512:
<ul>
<li>Output the repeated match with a REP packet</li>
</ul>
</li>
<li>If a repeated match was found, and its length is at least 3 character less than the main match, and the main match distance is at least 32768:
<ul>
<li>Output the repeated match with a REP packet</li>
</ul>
</li>
<li>If the main match size is less than 2 (or there isn't any match):
<ul>
<li>Output a LIT packet</li>
</ul>
</li>
<li>Perform a dictionary search for the next byte</li>
<li>If the next byte has a match with length at least one less than the main match length, with distance less than 1/128 times the main match distance, and if the main match length is at least 3:
<ul>
<li>Output a LIT packet</li>
</ul>
</li>
<li>If the next byte has a match at least as long as the main match, and with less distance than the main match:
<ul>
<li>Output a LIT packet</li>
</ul>
</li>
<li>If the next byte has a match at least one character longer than the main match, and such that 1/128 of its distance is less or equal than the main match distance:
<ul>
<li>Output a LIT packet</li>
</ul>
</li>
<li>If the next byte has a match more than one character longer than the main match:
<ul>
<li>Output a LIT packet</li>
</ul>
</li>
<li>If any repeated match has length at least one less than the main match length:
<ul>
<li>Output the most frequently used such distance with a REP packet</li>
</ul>
</li>
<li>Output the main match with a MATCH packet</li>
</ol>
<h4><span class="mw-headline" id="Normal_encoder">Normal encoder</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=18" title="Edit section: Normal encoder">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The XZ normal encoder<sup id="cite_ref-12" class="reference"><a href="#cite_note-12">[12]</a></sup> (derived from the 7-zip normal encoder) is the other LZMA encoder in the <a href="/wiki/Xz" title="Xz">xz</a> source tree, which adopts a more sophisticated approach that tries to minimize the post-range-encoding size of the generated packets.</p>
<p>Specifically, it encodes portions of the input using the result of a dynamic programming algorithm, where the subproblems are finding the approximately optimal encoding (the one with minimal post-range-encoding size) of the substring of length L starting at the byte being compressed.</p>
<p>The size of the portion of the input processed in the dynamic programming algorithm is determined to be the maximum between the longest dictionary match and the longest repeated match found at the start position (which is capped by the maximum LZMA match length, 273); furthermore, if a match longer than <i>nice_len</i> is found at any point in the range just defined, the dynamic programming algorithm stops, the solution for the subproblem up to that point is output, the <i>nice_len</i>-sized match is output, and a new dynamic programming problem instance is started at the byte after the match is output.</p>
<p>Subproblem candidate solutions are incrementally updated with candidate encodings, constructed taking the solution for a shorter substring of length L', extended with all possible "tails", or sets of 1-3 packets with certain constraints that encode the input at the L' position. Once the final solution of a subproblem is found, the LZMA state and least used distances for it are computed, and are then used to appropriately compute post-range-encoding sizes of its extensions.</p>
<p>At the end of the dynamic programming optimization, the whole optimal encoding of the longest substring considered is output, and encoding continues at the first uncompressed byte not already encoded, after updating the LZMA state and least used distances.</p>
<p>Each subproblem is extended by a packet sequence which we call "tail", which must match one of the following patterns:</p>
<table class="wikitable" border="1">
<tr>
<th>1st packet</th>
<th>2nd packet</th>
<th>3rd packet</th>
</tr>
<tr>
<td>any</td>
<td></td>
<td></td>
</tr>
<tr>
<td>LIT</td>
<td>LONGREP[0]</td>
<td></td>
</tr>
<tr>
<td>*MATCH</td>
<td>LIT</td>
<td>LONGREP[0]</td>
</tr>
</table>
<p>The reason for not only extending with single packets is that subproblems only have the substring length as the parameter for performance and algorithmic complexity reasons, while an optimal dynamic programming approach would also require to have the last used distances and LZMA <i>state</i> as parameter; thus, extending with multiple packets allows to better approximate the optimal solution, and specifically to make better use of LONGREP[0] packets.</p>
<p>The following data is stored for each subproblem (of course, the values stored are for the candidate solution with minimum <i>price</i>), where by "tail" we refer to the packets extending the solution of the smaller subproblem, which are described directly in the following structure:</p>
<table class="wikitable" border="1">
<tr>
<th>XZ for Java member name</th>
<th>description</th>
</tr>
<tr>
<td>price</td>
<td>quantity to be minimized: number of post-range-encoding bits needed to encode the string</td>
</tr>
<tr>
<td>optPrev</td>
<td>uncompressed size of the substring encoded by all packets except the last one</td>
</tr>
<tr>
<td>backPrev</td>
<td>-1 if the last packet is LIT, 0-3 if it is a repetition using the last used distance number 0-3, 4 + <i>distance</i> if it is a MATCH (this is always 0 if prev1IsLiteral is true, since the last packet can only be a LONGREP[0] in that case)</td>
</tr>
<tr>
<td>prev1IsLiteral</td>
<td>true if the "tail" contains more than one packet (in which case the one before the last is a LIT)</td>
</tr>
<tr>
<td>hasPrev2</td>
<td>true if the "tail" contains 3 packets (only valid if prev1IsLiteral is true)</td>
</tr>
<tr>
<td>optPrev2</td>
<td>uncompressed size of the substring encoded by all packets except the "tail" (only valid if prev1IsLiteral and hasPrev2 are true)</td>
</tr>
<tr>
<td>backPrev2</td>
<td>-1 if the first packet in the "tail" is LIT, 0-3 if it is a repetition using the last used distance number 0-3, 4 + <i>distance</i> if it is a MATCH (only valid if prev1IsLiteral and hasPrev2 are true)</td>
</tr>
<tr>
<td>reps[4]</td>
<td>the values of the 4 last used distances after the packets in the solution (computed only after the best subproblem solution has been determined)</td>
</tr>
<tr>
<td>state</td>
<td>the LZMA <i>state</i> value after the packets in the solution (computed only after the best subproblem solution has been determined)</td>
</tr>
</table>
<p>Note that in the XZ for Java implementation, the <i>optPrev</i> and <i>backPrev</i> members are reused to store a forward single-linked list of packets as part of outputting the final solution.</p>
<h3><span class="mw-headline" id="LZMA2_encoder">LZMA2 encoder</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=19" title="Edit section: LZMA2 encoder">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <a href="/wiki/XZ_Utils" title="XZ Utils">XZ</a> LZMA2 encoder processes the input in chunks (of up to 2 MB uncompressed size or 64 KB compressed size, whichever is lower), handing each chunk to the LZMA encoder, and then deciding whether to output an LZMA2 LZMA chunk including the encoded data, or to output an LZMA2 uncompressed chunk, depending on which is shorter (LZMA, like any other compressor, will necessarily expand rather than compress some kinds of data).</p>
<p>The LZMA state is reset only in the first block, if the caller requests a change of properties and every time a compressed chunk is output. The LZMA properties are changed only in the first block, or if the caller requests a change of properties. The dictionary is only reset in the first block.</p>
<h3><span class="mw-headline" id="Upper_encoding_layers">Upper encoding layers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=20" title="Edit section: Upper encoding layers">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Before LZMA2 encoding, depending on the options provided, <a href="/wiki/Xz" title="Xz">xz</a> can apply the BCJ filter, which filters executable code to replace relative offsets with absolute ones that are more repetitive, or the delta filter, which replaces each byte with the difference between it and the byte <var style="padding-right: 1px;">N</var> bytes before it.</p>
<p>Parallel encoding is performed by dividing the file in chunks which are distributed to threads, and ultimately each encoded (using, for instance, <a href="/wiki/Xz" title="Xz">xz</a> block encoding) separately, resulting in a dictionary reset between chunks in the output file.</p>
<h2><span class="mw-headline" id="7-Zip_reference_implementation">7-Zip reference implementation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=21" title="Edit section: 7-Zip reference implementation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The LZMA implementation extracted from <a href="/wiki/7-Zip" title="7-Zip">7-Zip</a> is available as LZMA SDK. It was originally dual-licensed under both the <a href="/wiki/GNU_Lesser_General_Public_License" title="GNU Lesser General Public License">GNU LGPL</a> and <a href="/wiki/Common_Public_License" title="Common Public License">Common Public License</a>,<sup id="cite_ref-13" class="reference"><a href="#cite_note-13">[13]</a></sup> with an additional special exception for linked binaries, but was placed by <a href="/wiki/Igor_Pavlov_(programmer)" title="Igor Pavlov (programmer)">Igor Pavlov</a> in the <a href="/wiki/Public_domain" title="Public domain">public domain</a> on December 2, 2008, with the release of version 4.62.<sup id="cite_ref-LZMA_SDK_8-1" class="reference"><a href="#cite_note-LZMA_SDK-8">[8]</a></sup></p>
<p>LZMA2 compression, which is an improved version of LZMA,<sup id="cite_ref-14" class="reference"><a href="#cite_note-14">[14]</a></sup>is now the default compression method for the .7z format, starting with version 9.30 on October 26th, 2012.<sup id="cite_ref-15" class="reference"><a href="#cite_note-15">[15]</a></sup></p>
<p>The reference <a href="/wiki/Open_source" class="mw-redirect" title="Open source">open source</a> LZMA compression library is written in <a href="/wiki/C%2B%2B" title="C++">C++</a> and has the following properties:</p>
<ul>
<li>Compression speed: approximately 1&#160;<a href="/wiki/Megabyte_per_second" class="mw-redirect" title="Megabyte per second">MB/s</a> on a 2&#160;<a href="/wiki/GHz" class="mw-redirect" title="GHz">GHz</a> <a href="/wiki/Central_processing_unit" title="Central processing unit">CPU</a>.</li>
<li>Decompression speed: 10–20&#160;MB/s on a 2&#160;GHz CPU.</li>
<li>Support for <a href="/wiki/Thread_(computer_science)" class="mw-redirect" title="Thread (computer science)">multithreading</a>.</li>
</ul>
<p>In addition to the original <a href="/wiki/C%2B%2B" title="C++">C++</a>, the LZMA SDK contains <a href="/wiki/Reference_implementation" title="Reference implementation">reference implementations</a> of LZMA compression and decompression ported to <a href="/wiki/C_(programming_language)" title="C (programming language)">ANSI C</a>, <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>, and <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>.<sup id="cite_ref-LZMA_SDK_8-2" class="reference"><a href="#cite_note-LZMA_SDK-8">[8]</a></sup> There are also third-party <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a> bindings for the C++ library, as well as ports of LZMA to <a href="/wiki/Pascal_(programming_language)" title="Pascal (programming language)">Pascal</a>, <a href="/wiki/Go_(programming_language)" title="Go (programming language)">Go</a> and <a href="/wiki/Ada_(programming_language)" title="Ada (programming language)">Ada</a>.<sup id="cite_ref-16" class="reference"><a href="#cite_note-16">[16]</a></sup><sup id="cite_ref-17" class="reference"><a href="#cite_note-17">[17]</a></sup><sup id="cite_ref-18" class="reference"><a href="#cite_note-18">[18]</a></sup><sup id="cite_ref-19" class="reference"><a href="#cite_note-19">[19]</a></sup></p>
<p>The 7-Zip implementation uses several variants of <a href="/wiki/Hash_chain" title="Hash chain">hash chains</a>, <a href="/wiki/Binary_tree" title="Binary tree">binary trees</a> and <a href="/wiki/Patricia_trie" class="mw-redirect" title="Patricia trie">Patricia tries</a> as the basis for its dictionary search algorithm.</p>
<p>Decompression-only code for LZMA generally compiles to around 5&#160;KB, and the amount of RAM required during decompression is principally determined by the size of the <a href="/wiki/Sliding_window" class="mw-redirect" title="Sliding window">sliding window</a> used during compression. Small code size and relatively low memory overhead, particularly with smaller dictionary lengths, and free source code make the LZMA decompression algorithm well-suited to <a href="/wiki/Embedded_system" title="Embedded system">embedded</a> applications.</p>
<h2><span class="mw-headline" id="Other_implementations">Other implementations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=22" title="Edit section: Other implementations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In addition to the 7-Zip reference implementation, the following support the LZMA format.</p>
<ul>
<li><a href="/wiki/Xz" title="Xz">xz</a>: a streaming implementation that contains a <a href="/wiki/Gzip" title="Gzip">gzip</a>-like command line tool, supporting both LZMA and LZMA2 in its xz file format. It made its way into several software of the <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> world with its high performance (compared to <a href="/wiki/Bzip2" title="Bzip2">bzip2</a>) and small size (compared to <a href="/wiki/Gzip" title="Gzip">gzip</a>).<sup id="cite_ref-gzip_bzip_lzma_bench_2-1" class="reference"><a href="#cite_note-gzip_bzip_lzma_bench-2">[2]</a></sup> The <a href="/wiki/Linux_kernel" title="Linux kernel">Linux kernel</a>, <a href="/wiki/Dpkg" title="Dpkg">dpkg</a> and <a href="/wiki/RPM_Package_Manager" class="mw-redirect" title="RPM Package Manager">RPM</a> systems contains xz code, and many software distributors like <a href="/wiki/Linux_kernel" title="Linux kernel">kernel.org</a>, <a href="/wiki/Debian" title="Debian">Debian</a><sup id="cite_ref-dpkg_use_xz_20-0" class="reference"><a href="#cite_note-dpkg_use_xz-20">[20]</a></sup> and <a href="/wiki/Fedora" title="Fedora">Fedora</a> now use xz for compressing their releases.</li>
<li><a href="/wiki/Lzip" title="Lzip">lzip</a>: another LZMA implementation mostly for <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> systems to be directly competing with <a href="/wiki/Xz" title="Xz">xz</a>.<sup id="cite_ref-lzip_bench_21-0" class="reference"><a href="#cite_note-lzip_bench-21">[21]</a></sup> It mainly features a simpler file format and therefore easier error recovery.</li>
<li><a href="/wiki/Zip_(file_format)" title="Zip (file format)">ZIPX</a>: an extension to the <a href="/wiki/Zip_(file_format)" title="Zip (file format)">ZIP</a> compressions format that was created by <a href="/wiki/WinZip" title="WinZip">WinZip</a> starting with version 12.1. It also can use various other compression methods such as <a href="/wiki/Bzip2" title="Bzip2">BZip</a> and <a href="/wiki/Prediction_by_partial_matching" title="Prediction by partial matching">PPMd</a>. <sup id="cite_ref-zipx_22-0" class="reference"><a href="#cite_note-zipx-22">[22]</a></sup></li>
<li><a rel="nofollow" class="external text" href="https://www.noemax.com/dotnetcompression/">DotNetCompression</a>: a streaming implementation of LZMA in managed C#.</li>
</ul>
<h2><span class="mw-headline" id="LZHAM">LZHAM</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm&amp;action=edit&amp;section=23" title="Edit section: LZHAM">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>LZHAM (LZ, Huffman, Arithmetic, Markov), is an LZMA-like implementation that trades compression throughput for very high ratios and higher decompression throughput.<sup id="cite_ref-23" class="reference"><a href="#cite_note-23">[23]</a></sup></p>
